## 背景与目标
纯SMP架构下的Linux难以满足us级的硬实时需求,引入物理AMP虽可弥补,但会带来额外的硬件成本与系统复杂度。
为此,本项目提出了一种在不增加硬件的前提下,基于软件分域将同构多核构造为AMP系统的整体方案。

本项目在一颗**同构多核RISC-V SoC**上，通过OpenSBI 的Domain机制和PMP将物理核划分为两个隔离域：一个域运行RTOS（承担实时控制任务），另一个域运行Linux（承担UI、网络、复杂业务等）。

这本质上是用**软件手段+硬件权限管理构造出的AMP（非对称多处理）系统**，区别于传统由异构核（如Cortex-A + Cortex-M）组成的**物理AMP系统**。

**核心命题**：物理AMP的**隔离性**、**实时性**、**安全性**和**低功耗**大多是“物理分离”天然带来的；软件分域必须用软件配合共享硬件的QoS机制，把这些属性重新“造”出来。因此本方案的目标不是简单地把两个系统跑起来，而是**逐项磨平软件分域相对物理AMP的差距，尽量向真正的硬件AMP靠拢，达到工程可用的实时整合系统**。

## 物理AMP系统的特性分析

以主流物理AMP平台（如NXP i.MX、TI Sitara、ST STM32MP等A核 + M核架构）为参照，其关键特性来源于**物理分离**：

| 维度 | 物理AMP的实现方式 | 特性来源 |
| --- | --- | --- |
| 内存空间隔离 | 独立地址空间 + 资源域控制器（RDC/Firewall） | 硬件强制 |
| DMA/外设IO隔离 | 总线主设备受资源域控制器约束 | 硬件强制 |
| 实时性 | M核带TCM、独立/简单cache，不与A核抢内存带宽 | 物理独占 |
| 中断确定性 | 独立或专用中断控制器，路由干净 | 物理分离 |
| 故障隔离 | A核崩溃不影响M核，M核可反向拉起A核 | 物理独立 |
| 独立生命周期 | 各核独立启动、复位、休眠 | 物理独立 |
| 核间通信 | 底层共享内存 + Mailbox/IPI，上层RPMsg/virtio | 标准机制 |
| 电源管理 | 独立电源域，M核可always-on微安级常驻 | 独立电源域 |

**结论**：物理AMP的绝大部分优势，是“两块相互独立的硬件”这个前提免费赠送的。软件分域没有这个前提，必须逐项补齐。

## 软件分域方案的能力现状与差距

在同构RISC-V核 + OpenSBI上，Domain是“核+带权限内存区+启动入口”的隔离描述, 当前能直接获得和缺失的能力如下：

**已具备能力（OpenSBI Domain原生提供）:**


- 静态内存划分与启动隔离（基于硬件PMP/ePMP + 软件opensbi配置）
- 各域独立入口、独立引导（软件opensbi配置）

**和硬件AMP存在本质差距的地方:**

- 内存隔离：两域共享LLC、内存控制器、总线，Linux侧负载会冲刷cache、吃满DDR带宽，导致RTOS侧中断延迟与执行时间剧烈抖动。**这是软件分域最容易翻车、也最能拉开与物理AMP差距的一层。**
- DMA隔离：PMP只约束CPU访存，不管外设DMA，隔离存在被击穿风险。
- 故障隔离与恢复：Domain提供了启动隔离，但“崩溃后互不影响 + 可恢复”需自行在M态补齐。
- 低功耗：满血应用核即使门控降频，静态功耗与唤醒代价也远高于专用小核。

## 逐项对比与追平手段（方案核心）

按能力维度逐项拆解：物理AMP如何做、软件分域的问题、以及本项目的磨平手段。

| 维度 | 物理AMP | 软件分域问题 | 追平手段 |
| --- | --- | --- | --- |
| CPU访存隔离 | 资源域控制器硬件强制 | PMP/ePMP可切分地址空间，可达标 | 用**ePMP（Smepmp）**增强M态隔离约束，静态配置各域PMP区域禁止跨域访问。**可基本追平** |
| IO访存隔离 | 总线主设备DMA/外设受RDC/Firewall约束 | PMP只约束CPU访存，管不住DMA主设备，Linux驱动的DMA可越界写入RTOS内存，属“纸面隔离” | 引入**IOPMP**（或SiFive WorldGuard类基于RCID的世界隔离）约束所有总线主设备。**无IOPMP则安全隔离不成立**，需在选型阶段确认 |
| 访存干扰隔离 | M核带TCM、物理独占，天然确定 | 共享LLC与内存带宽，Linux负载可能导致RTOS抖动 | **Ssqosid（RCID/MCID）+ CBQRI**做cache way/带宽分区；<br/>核带**TCM/Scratchpad**则放RTOS关键路径代码与数据；<br/>DDR控制器**QoS**配per-core流量优先级；<br/>无硬件支持则退到软件**MemGuard** |
| 中断确定性 | 独立/专用中断控制器 | PLIC共享，路由路径过长可能导致实时性不足 | **APLIC+IMSIC**用MSI将中断直投指定hart，低延迟且天然按域隔离；<br/>或配PLIC的S模式委托与路由投递，减少路由次数 |
| 定时确定性 | 各核独立定时器 | 时钟中断经M态陷入走SBI调用，引入抖动 | **Sstc**扩展，让RTOS域直接读写stimecmp，避免每次陷入M态 |
| 故障隔离与独立生命周期 | A核崩溃不影响M核，可反向恢复 | Domain仅提供启动隔离，崩溃互不影响与可恢复需自行实现 | 每域**独立启动/复位/恢复**，一侧panic不拖垮另一侧；<br/>**看门狗 + 故障封闭（RAS）**，总线/ECC错误不跨域传播；<br/>RTOS域作监护者，检测并重启Linux域（对标remoteproc反向拉起） |
| 核间通信 | 共享内存 + Mailbox/IPI + RPMsg/virtio | 此层两种架构差异不大 | 共享内存 + IPI/mailbox + 无锁环形缓冲，上层跑**RPMsg/virtio（OpenAMP）**。**可基本追平**，应用侧几乎无感知差异 |
| 电源管理 | 独立电源域，M核可always-on | 满血核低功耗能力有限 | 尝试**每核独立DVFS与power gating**，让RTOS域低功耗常驻、Linux域深睡。**此项难以完全追平**（见下一节） |

## 无法完全磨平的边界

即便上述能力全部补齐，以下几点是“软件构造”难以追平物理分离的：

- **低功耗**：物理AMP可用微安级小核做always-on；满血RISC-V应用核即使门控降频，静态功耗与唤醒代价仍远高于此。**超低功耗always-on场景，软件分域基本无法追平。**
- **共享单点资源**：LLC、内存控制器仍是物理共享点，QoS能缓解干扰但无法消除。

## 能力目标定义与达成分级

建议将项目目标定位为**混合关键性整合（Mixed-Criticality）**，而非“完全等价物理AMP”。分级达成：

| 等级 | 能力要求 | 适用场景 |
| --- | --- | --- |
| L1软整合 | Domain隔离 + 核间通信 | 功能整合，无实时保证 |
| L2软实时 | IOPMP + AIA + Sstc + 独立恢复 | Linux管界面/网络，RTOS管一般控制 |
| L3准硬实时 | + Ssqosid/CBQRI cache/带宽分区（或TCM） | 电机控制、控制回路等确定性要求高的场景 |
| 物理AMP专属 | 微秒级零抖动硬实时 / 微安级always-on | 本方案不覆盖，仍需物理异构 |

**本项目目标**：达成**L3**，即具备**IOPMP + Ssqosid/CBQRI + AIA + Sstc + 域级独立复位恢复**的完整能力栈，做到对绝大多数“Linux管人机界面/网络、RTOS管控制回路”的场景足够接近物理AMP、工程真正可用。

## 选型建议

**价值定位**：本方案的并不能完全媲美物理AMP，而是**用一颗同构多核SoC吃下需要A+M双芯或双核簇才能承担的混合关键性负载**——节省物理核与BOM成本，代价是需要靠一整套隔离与QoS能力去逼近物理AMP。

**选型关键项**：

1. **IOPMP**（或等价世界隔离）——缺失则DMA隔离不成立，安全性不达标。
2. **Ssqosid + CBQRI**（cache/带宽分区）或**TCM**——缺失则只能做到L2软实时。
3. **AIA（APLIC + IMSIC）**——保证中断确定性。
4. **Sstc**——降低时钟中断抖动。
5. **ePMP（Smepmp）**与**OpenSBI Domain**支持——隔离与启动的基础。

**明确不覆盖的场景**：单微秒级零抖动硬实时、微安级always-on，仍需物理异构方案。
